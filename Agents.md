# 补丁仓库处理规范

## 仓库边界

- 当前仓库只用于维护补丁，不包含上游源码。
- 禁止将上游源码目录添加、暂存或提交到当前补丁仓库。
- 如果将上游仓库克隆到当前仓库下的 `new-api/`，该目录必须始终保持未跟踪状态。
- 保留当前仓库及已有上游副本中与任务无关的修改。

## 上游代码准备

- 每次处理补丁前，必须主动从 `https://github.com/QuantumNous/new-api` 获取最新上游代码，不得等待用户预先准备 `new-api/` 目录。
- 可以将上游仓库克隆到当前仓库下的 `new-api/`，也可以使用仓库外的任意独立临时目录。
- 如果已存在上游副本，仍必须先获取远端最新状态并以远端最新 HEAD 为验证基线；不得默认本地副本已经是最新版本。
- 不得覆盖或清理存在未提交修改的上游副本。遇到这种情况，应另建临时目录并重新克隆。
- 补丁验证必须在可丢弃的干净副本中进行，不得在需要保留的上游副本中留下补丁应用结果。

## 任务目标

- 按数字序号依次检查 `patches/*.patch`。
- 每个补丁必须先通过 `git apply --check`，再实际应用，然后才能检查下一个补丁。
- 只有全部补丁在同一个干净上游副本中依次应用成功，才能判定任务完成。
- 不执行构建、编译、打包或发布流程。

## 冲突处理

### 仅上下文漂移

上下文漂移包括代码行移动、文件或函数改名，以及不影响补丁行为的上游重构。

- 直接将补丁 hunk 重新定位到当前上游的等价位置。
- 只调整路径、上下文、hunk 头和 index hash，不得改变补丁语义。
- 不得借机重构、扩展或重新设计功能。
- 修正后继续按序检查后续补丁，直到完整序列通过。

### 语义或功能冲突

出现以下任一情况时，立即停止，不得为该冲突修改补丁：

- 上游已经实现了补丁功能。
- 上游行为与补丁预期存在逻辑冲突。
- 合入需要产品、接口、数据模型或架构决策。
- 目标功能已经删除，且不存在明确的等价位置。
- 只有改变补丁语义才能继续合入。

报告第一个阻断补丁、失败文件或 hunk、相关上游改动及具体逻辑冲突。

## 验证要求

- 修改补丁后，必须重新创建干净上游副本，并从 `0001` 开始完整验证。
- 所有补丁必须同时通过检查和实际顺序应用，不能只单独检查失败补丁。
- 对最终补丁结果执行 `git diff --check`。
- 如果使用了当前仓库下的 `new-api/`，确认父仓库仍将其视为未跟踪目录，且其中没有遗留补丁应用结果。
- 不得在完整验证通过前宣称补丁可以合入。

## 提交要求

- 仅当全部补丁成功应用，且修改仅属于上下文漂移时才允许提交。
- 必须显式列出暂存文件，禁止使用 `git add .` 或 `git add -A`。
- 提交前检查 `git diff --cached --name-only`，确保不存在任何 `new-api/` 路径。
- 遵循仓库已有提交风格，例如：`fix(patches): refresh contexts for latest upstream`。
- 如果 `.git` 不可写，保留已验证的工作区修改，并报告错误和供用户执行的提交命令。

## 最终报告

说明上游代码的获取方式、已检查的上游 commit、全部补丁是否按序成功、调整了哪些补丁及原因，并提供 commit hash。明确说明未执行构建，且未将任何上游源码目录纳入当前仓库。
